# はじめに

---

## このハンズオンのゴール

> エージェント型の生成 AI を使って、文書作成の全工程を実践する力を身につける

- 「チャットで聞く」から「生成 AI と一緒に作業する」へ
- 生成 AI にはチャット型とエージェント型がある
  - チャット型 — チャット画面でやり取りし、回答をコピー&ペーストして使う
  - エージェント型 — 生成 AI が自らファイルを読み書きし、ツールを実行して作業を進める
- Claude Code はエージェント型のツール

---

## このハンズオンの趣旨

> 「生成 AI に任せる」ではなく「生成 AI と協業する」

- エージェント型生成 AI と協業するには、次の2点が重要
  - 作業工程を分解し、「今何をやっているか」を明確に
  - 各工程の結果は文書化する
- 各工程でエージェントにどう指示し、どう情報を共有し、どう協働するかを体験する

### 3 つの柱

1. 工程を踏む — 一発で完成させず、段階的に仕上げる
2. 人間が主導する — 生成 AI は道具。何を作るか決めるのは自分
3. ツールを組み合わせる — 小さなツールを Claude Code に使わせる

---

## このハンズオンの前提

- プログラミングの知識は不要
- 以下が準備済みであること
  - PC の基本操作ができる（ファイル操作、テキスト入力）
  - Markdown の基本的な書き方を学習済み（参考: [Markdown 記法一覧](https://gist.github.com/mignonstyle/083c9e1651d7734f84c99b8cf49d57fa)）
  - Git Bash が利用できる（Git for Windows に付属。インストール方法は後述）
  - Visual Studio Code がインストール済み
  - Claude Code がインストール済み
  - Claude Code が使用できる Claude のプランに契約しているか、API のクレジットを持っている

---

## Claude Code とは

> チャット型の生成 AI が賢い電話相談なら、エージェント型は隣の席で働くよくできた部下

- Anthropic 社が提供するエージェント型の生成 AI ツール — 回答をするだけでなく、人間に代わって作業を委任できる
- チャット型が得意でない次のことが可能
  - ファイルの読み書き
    - チャット型では不可能な大量の情報の受け渡しが可能
    - 直接のアウトプット作成が可能
  - コマンドの実行
    - ツールを直接操作する
    - ツール実行結果に基づいて自律的に次の作業を判断する

---

## Claude Code を使う上での注意

- **生成 AI は間違えることがある** — もっともらしい嘘（ハルシネーション）を生成することがある。その救済措置として、収集した情報や何をインプットにしたか、判断基準は何だったかを常に Claude Code に問い、ファイルに記録する癖をつける
- **費用が発生する** — Claude Code の利用には Claude Pro/Max プランの契約、または API クレジットが必要。利用量に応じて費用がかかるため、契約プランの内容を事前に確認しておくこと
- **入力内容はサーバーに送信される** — Claude Code で扱うファイルや入力テキストは Anthropic のサーバーに送信される。機密情報や個人情報を含むファイルを扱う際は注意すること

---

## AI 生成物と著作権

- 生成 AI が出力した文章は、そのままでは著作権で保護されない可能性がある
- 人間が構成を考え、指示を出し、取捨選択や修正を加えることで、人間の創作的寄与が認められやすくなる
- このハンズオンで「工程を踏む」「人間が主導する」ことを重視しているのは、品質だけでなく著作権の観点からも意味がある
- 詳しくは [文化審議会「AI と著作権に関する考え方について」（2024 年 3 月）](https://www.bunka.go.jp/seisaku/bunkashingikai/chosakuken/pdf/94037901_01.pdf) を参照

---

## 進め方

- 自分のペースで順番に進めるセルフハンズオン形式
- 全体の流れ（合計: 約 3 時間 40 分）
  1. 環境を整える（40 分）
  2. 構想する（60 分）
  3. 仕様を決める（60 分）
  4. 執筆する（30 分）
  5. 校正する（20 分）
  6. リリースする（10 分）
- つまずいたら Claude Code に「エラーが出た」と伝えれば対処してくれる

---

## 演習課題の概要

> **架空の経営支援サービス会社の社員として、Web サイトリニューアルの 提案依頼書(RFP) を作成する**

- システムのRFP を題材にする理由
  - 構造が明確 — 背景・目的・要件・評価基準など、決まったセクションがある
  - 実務に近い — 非エンジニアでも業務で触れる機会がある
  - 生成 AI との相性が良い — 要件の洗い出しや表現の推敲に向いている

---

## 文書作成の 5 つの工程

> 良い文書は「一発」では生まれない。工程を踏むから質が上がる

今回は、一般的に文書策定によいとされる次の工程で進めます。

| 工程     | やること                       | Claude Code の役割               |
| -------- | ------------------------------ | -------------------------------- |
| 構想     | 目的・読者・伝えたいことを整理 | 対話でアイデアを壁打ちする       |
| 仕様     | 構成・見出し・要件を決める     | 仕様書を作成する                 |
| 執筆     | 仕様に沿って本文を書く         | セクションごとに下書きを生成する |
| 校正     | 文章の品質をチェックする       | 校正ツールを実行し修正する       |
| リリース | 最終確認し成果物を仕上げる     | HTML 変換を行う                  |

# 1. 環境を整える（目安: 40 分）

完了の目安: Claude Code が起動でき、作業ディレクトリと校正ツールの準備ができている

---

## Git for Windows をインストールする

- Claude Code は内部で Unix 系のコマンドを使うため、Unix 互換のシェル（ターミナル）が必要
- Windows では Git for Windows に付属する Git Bash を使う
- 以下の URL からダウンロードしてインストールする
  - <https://git-scm.com/downloads/win>
- インストール時の設定はすべてデフォルトのままでよい

---

## Git Bash を起動する

- スタートメニューで「Git Bash」と検索して起動する
- 黒い画面にコマンド入力待ちの表示（`$`）が出れば OK
- 以降の作業はすべてこの Git Bash 上で行う

> Windows 標準の PowerShell やコマンドプロンプトでは Claude Code は動作しない。必ず Git Bash を使うこと

---

## ハンズオン資料の確認

- ダウンロード済みのハンズオン資料フォルダの中身を確認する

```
handson.md              # このスライド資料
resources/
  textlintrc.json       # 校正ツール（textlint）の設定ファイル
```

- `resources/textlintrc.json` は校正フェーズで使用する。後のステップでコピーするので、今は場所だけ覚えておく

---

## Claude Code をインストールする

- このステップだけ PowerShell を使う（スタートメニューで「PowerShell」と検索して起動）
- 以下のコマンドを入力して Enter キーを押す

```powershell
irm https://claude.ai/install.ps1 | iex
```

- インストールが完了するまで数分かかることがある
- インストール後は自動的に最新版に更新される
- インストールが終わったら PowerShell は閉じてよい
- 参考: [Claude Code セットアップガイド（公式）](https://docs.anthropic.com/en/docs/claude-code/setup)
- PowerShell で Claude Code は起動しない。次のステップで Git Bash から起動する

---

## Claude Code を起動する

- Git Bash で以下のコマンドを入力する

```bash
claude
```

- 初回起動時はログイン認証が求められる
  - ブラウザが自動的に開くので、Claude のアカウントでログインする
- 認証後、初回の設定画面が表示される
  - テーマや権限の設定を聞かれるので、案内に従って選択する
- 設定が完了すると対話画面が表示される
- 試しに「こんにちは」と入力して応答を確認する
- 終了するには `/exit` と入力する

---

## 作業ディレクトリを作成する

- これから作成するファイルを置く場所を用意する
- まずデスクトップに移動し、フォルダ構成を作成する

```bash
cd ~/Desktop
mkdir rfp-project
cd rfp-project
mkdir input docs src output
```

- フォルダの役割
  - `input/` — インプット資料（手持ちのメモや参考資料）
  - `docs/` — 検討資料（構想・仕様フェーズの成果物）
  - `src/` — 成果物ソース（RFP 本体の Markdown）
  - `output/` — 変換済み出力（pandoc で生成した HTML）
- 以降はこのフォルダの中で Claude Code を起動して作業する

---

## Visual Studio Code をインストールする

- Visual Studio Code（以降 VS Code）は Microsoft 製の無料テキストエディタ
- Claude Code が作成・編集したファイルを確認するために使う
- 以下の URL からダウンロードしてインストールする
  - <https://code.visualstudio.com/>
- 拡張機能「Markdown Preview Enhanced」もインストールする
  - Markdown をブラウザでリアルタイムにプレビューできる
  - VS Code の拡張機能タブで「Markdown Preview Enhanced」を検索してインストール
- Git Bash からも起動できる

```bash
code .
```

---

## Node.js をインストールする

- 校正ツール（textlint, markdownlint）の動作環境として Node.js が必要
- 以下の URL から LTS 版をダウンロードしてインストールする
  - <https://nodejs.org/>
- インストール後、Git Bash を再起動して以下で確認する

```bash
node --version
```

- バージョン番号が表示されれば OK

---

## markdownlint のインストール

- markdownlint は Markdown の構造と書式の一貫性をチェックするツール
- Claude Code に指示してインストールする

```
markdownlint-cli2 をインストールしてください
```

- Claude Code が環境に応じたコマンドを判断し、実行してくれる

---

## textlint のインストール

- textlint は日本語の文章品質をチェックするツール（冗長表現、表記ゆれなど）

```bash
cp resources/textlintrc.json .textlintrc

```

---

## 校正ツールの情報を記録する

- インストールした校正ツールの情報を CLAUDE.md に記録しておく

```
校正ツール（textlint, markdownlint）の情報を CLAUDE.md に記録してください。
```

---

## 変換ツールのインストール

- 完成した文書を最終成果物（HTML など）に変換するために pandoc を使う
- pandoc — Markdown を HTML や PDF など様々な形式に変換するツール
- Claude Code に指示してインストールする

```
pandoc をインストールしてください
```

- Claude Code が環境に応じたコマンドを判断し、実行してくれる

---

## Magi プラグインとは

> エヴァンゲリオンの MAGI システムに着想を得た Claude Code の意思決定支援プラグイン

- 3 つの専門家視点で分析し、多数決で結論を出す
  - MELCHIOR — 科学・技術面の分析
  - BALTHASAR — 法務・倫理面の分析
  - CASPER — 感情・トレンドの分析
- 今回は文書のレビューに活用する
- 筆者が Claude Code で作成したプラグイン。コードは一行も書いていない
  - Claude Code のプラグインは Markdown（文章）だけで作れる

---

## Magi のインストール手順

1. Claude Code で `/plugin` と入力する
2. **Marketplaces** タブ → **+ Add Marketplace** を選ぶ
3. 以下を入力して Enter キーを押す
   - `https://github.com/yostos/claude-code-plugins`
4. 再度 `/plugin` → **Marketplaces** タブを開き、追加されたマーケットプレイスを選択する
5. **Browse Plugins** を選ぶと Magi が表示される
6. **Install plugin** を選んでインストールする
7. Claude Code を再起動すると `/magi` コマンドが使えるようになる

# 2. 構想する（目安: 60 分）

完了の目安: `docs/concept.md` が作成され、CLAUDE.md に記録されている

---

## 構想フェーズでやること

> 今回はテーマが決まっているので生成 AI に相談すれば適切な案が出てくる。しかし、文書の基本方針を Claude Code を使って自分で検討し明確にしておくことが重要

- まず手持ちの素材（ネタ）を Claude Code に読み込ませる
- その上で以下の 3 点を明確にする
  - どんな文書を書くのか（RFP とはどういう文書か）
  - 誰に向けて書くのか（この RFP の読み手は誰か）
  - 何のために書くのか（この RFP で達成したいことは何か）
- ここで決めたことが、後続フェーズ（仕様・執筆）での判断基準になる

---

## ネタを用意する

> 構想を始める前に、手持ちの情報を整理しておく

- RFP に関連する素材を `input/` フォルダに置く
  - 組織の概要、事業内容 / 現行サイト評価 / 再構築の目的や期待効果 / 予算・スケジュール目安
- `input/index.md` を作成し、Claude Codeへのヒントとして各資料の意味を書く

`index.md例`

```markdown
# インプット資料一覧

- https://hogehoge.hagehage.org/about : 会社概要と事業内容
- current-site-issues.md : 現行サイトの課題リスト
- budget-memo.xlsx : 予算とスケジュールの概算
```

---

## どんな文書を書くのか

- 経営支援サービス会社の Web サイトリニューアルに関する RFP を書く
- Claude Code に伝える

```
経営支援サービス会社が Web サイトをリニューアルするための
RFP（提案依頼書）を作成します。
```

- Claude Code がプロジェクトの文脈を理解し、以降の作業に活かしてくれる

---

## 誰に向けて書くのか

- 続けて Claude Code に指示する

```
この RFP を読む人は誰ですか？読者を整理してください。
```

- RFP の読者（提案を提出する側の企業）を明確にする
- 読者が明確になると、書くべき情報の粒度や表現が決まる

---

## 何のために書くのか

- 続けて Claude Code に指示する

```
この RFP は何のために書くのか、目的を整理してください。
背景として、架空の経営支援サービス会社が Web サイトをリニューアルしたいと
考えています。
```

- Claude Code が目的の案を提示してくれる
- 内容を確認し、修正や追加を指示する

---

## concept.md を作成する

- 構想の成果物を文書として残す。これが後続フェーズの判断基準になる
- ここまでの構想をファイルとして保存する

```
ここまでの議論を concept.md にまとめてください。
どんな文書か、読者、目的を整理して書いてください。
```

- Claude Code がファイルを作成し、VS Code で確認できる
- 内容を読んで、修正があれば Claude Code に指示する

---

## CLAUDE.md に記録する

- CLAUDE.md はプロジェクトの情報を Claude Code に共有するためのファイル
- 構想の成果物（concept.md）の位置づけを記録しておく

```
CLAUDE.md を作成して、このプロジェクトの概要と
concept.md の役割を記録してください。
```

- 次回 Claude Code を起動したとき、CLAUDE.md を読んでプロジェクトの文脈を理解してくれる

---

# 3. 仕様を決める（目安: 60 分）

完了の目安: `docs/requirements.md` が作成され、CLAUDE.md に記録されている

---

## 仕様フェーズでやること

> 構想で決めた制約をもとに、RFP の文書要件と構成を決める

1. RFP の構成要素を調査する
   - RFP の標準的な構成要素を調査する
2. 文書要件を整理する
   - 調査結果を参考に、文書要件項目と定義の一覧にまとめる
3. 構成を決める
   - 構想の制約を判断基準に、要件項目の取捨選択と並び順を決める
4. ツールとフォーマットを選択する
   - Markdown で執筆し、pandoc で HTML に変換する方針を決める
   - 校正ルールを決める

- 各ステップの成果は情報源とともに requirements.md に記録していく

---

## RFP の構成要素を調査する

- Claude Code に RFP の標準的な構成要素を調べてもらう

```
構想文書（concept.md）に従って、
システム開発の RFP に必要な構成要素を調査してください。
情報源も示してください。
```

- Claude Code が Web 検索や知識をもとに構成要素を提示してくれる
- 複数の情報源を比較することで、信頼性を確認する
- 調査結果は情報源とともに requirements.md に記録しておく
  - 後で振り返るときに、なぜその構成にしたかを確認できる

---

## 文書要件を整理する

- 調査結果をもとに、文書要件としてまとめる

```
調査結果をもとに、RFP の文書要件項目とその定義を一覧にしてください。
```

- 各項目が何を意味するのか、定義を明確にしておく
- この一覧が次のステップで取捨選択する材料になる

---

## 構成を決める

- concept.md の制約を判断基準に、要件項目の取捨選択と並び順を決める

```
concept.md を読んで、この RFP に必要な項目を選び、
並び順を決めてください。不要な項目があれば理由も説明してください。
```

- Claude Code の提案をそのまま受け入れず、自分で判断する
- 構想で決めた目的・読者に照らして「この項目は必要か？」を考える

---

## ツールとフォーマットを選択する

- 今回は Markdown で執筆し、pandoc で HTML に変換する
- Claude Code に記録してもらう

```
文書は Markdown で執筆し、pandoc で HTML に変換する方針を
CLAUDE.md に記録してください。
```

- ツールの選定も仕様の一部。後から迷わないよう明文化しておく

---

## requirements.md を作成する

- 仕様の成果物を文書として残す。これが執筆フェーズの設計図になる

```
ここまでの仕様を requirements.md にまとめてください。
文書要件、構成、ツール選定を整理して書いてください。
```

- Claude Code がファイルを作成し、VS Code で確認できる
- 内容を読んで、修正があれば Claude Code に指示する
- requirements.md の役割も CLAUDE.md に記録しておく

```
requirements.md の役割を CLAUDE.md に記録してください。
```

---

# 4. 執筆する（目安: 30 分）

完了の目安: `src/rfp.md` が完成し、Magi レビューの指摘を反映済み

---

## 執筆フェーズでやること

> 仕様で決めた構成に沿って、Claude Code と一緒に RFP の本文を書いていく

1. Claude Code で執筆する — セクションごとにドラフトを作成する
2. VS Code で内容を確認する — Preview を使ってドラフトを確認する
3. Claude Code に修正を依頼する — ドラフトに対して必要な修正を依頼する
4. Magi でレビューする — 複数の観点から文書をチェックする
5. レビュー指摘を改修する — 指摘事項を Claude Code で修正する

---

## Claude Code で執筆する

> セクションごとにドラフトを作成する。一気に全体を書かせない

- requirements.md の構成に従い、セクションごとに Claude Code に指示する

```
requirements.md の構成に従って、
RFP の「背景と目的」セクションを src/rfp.md に
書いてください。input/ の資料を参考にしてください。
```

- 1 セクションずつドラフトを作成し、次のステップで確認する

---

## VS Code で内容を確認する

> ドラフトは必ず自分の目で確認する。生成されたままにしない

- VS Code で `src/rfp.md` を開き、Markdown Preview で確認する
- 確認のポイント
  - 内容に事実と異なる記述がないか
  - 読者にとって分かりにくい表現はないか
  - 各セクションの分量のバランスはよいか
- 気になる箇所をメモしておく

---

## Claude Code に修正を依頼する

> 修正したい点を具体的に伝えるほど、意図通りの結果が得られる

- 確認で気づいた点を Claude Code に伝えて修正する

```
src/rfp.md の「背景と目的」セクションを修正してください。
- 現行サイトの課題をもっと具体的に書いてください
- 文章が長いので簡潔にしてください
```

- 明確な誤りでなくても、納得できない箇所は「なぜそう書いたのか」を聞いてみる
  - 理由を聞くことで、自分の理解が深まったり、より良い表現が見つかることがある
- 修正結果を VS Code で再確認する
- 納得できたら次のセクションに進む
- すべてのセクションが書けたら、レビューに進む

---

## Magi でレビューする

> Magi は環境準備でインストールしたプラグイン。3 人の専門家が同時にレビューしてくれる

- Claude Code で `/magi` コマンドを実行する

```
src/rfp.md をレビューしてください。
前提事項は docs/ に、関連資料は input/ にあります。
```

- 3 つの観点からの指摘が返ってくるので、指摘事項をファイルに保存しておく
  - 票が割れた場合、Magi は討議制に移行するかどうかを確認してくる。この機能は Claude Max プランが必要

```
レビュー結果を docs/inspection-log.md に保存してください
```

- 合否よりも指摘事項が重要。すべての指摘を採用する必要はなく自分で判断する

---

## レビュー指摘を改修する

> レビュー指摘は「バグ報告」。直すかどうかは自分で判断する

- Magi のレビュー結果を確認し、対応する指摘を選ぶ
- Claude Code に修正を指示する

```
Magi のレビューで指摘された以下の点を修正してください。
- [修正したい指摘事項]
```

- 修正後、VS Code で再確認する
- 必要に応じてレビュー → 修正を繰り返す

---

# 5. 校正する（目安: 20 分）

完了の目安: markdownlint と textlint の指摘が 0 件になっている

---

## 校正フェーズでやること

> Magi のレビューは内容面のチェック。校正は文章と書式の機械的なチェック

- 2 つの校正ツールを Claude Code 経由で実行する
  - markdownlint — Markdown の構造と書式をチェック（見出し構造、箇条書き、空白ルールなど）
  - textlint — 日本語の文章品質をチェック（冗長表現、表記ゆれなど）
- ツールごとに役割が違う。両方実行して初めて品質が担保される
- 指摘を受け入れるかどうかは自分で判断する

---

## markdownlint を実行・修正する

> markdownlint は Markdown の構造と書式の一貫性をチェックするツール

- 構造的な修正なので、実行と修正を一度に指示してよい

```
src/rfp.md に対して markdownlint を実行し、指摘事項を修正してください
```

- 修正される例
  - 見出しレベルの飛び（h1 の次に h3 が来ているなど）
  - 見出しやコードブロックの前後に空行がない
  - 箇条書きの記号やインデントが統一されていない
- 修正後、VS Code の Markdown Preview で見た目が想定通りか確認する

---

## textlint を実行する

> textlint は日本語の文章品質をチェックするツール

- 続けて Claude Code に指示する

```
src/rfp.md に対して textlint を実行してください
```

- 指摘の例
  - 「することができる」→「できる」（冗長表現）
  - 「行なう」→「行う」（表記ゆれ）
  - 一文が長すぎる

---

## textlint の指摘を修正する

> 一括で修正を任せず、指摘ごとに修正方針を相談する

- 助詞の連続や読点の多さなど、Claude Code が対応に困る指摘がある
  - どう直すかを自分で判断し、具体的に指示する

```
textlint の指摘について、修正しましょう。
修正方法は一つ一つ順番に相談してください。
```

- 提案された修正案を確認し、納得できたら適用を指示する

---

## 再チェックする

> 修正によって新たな問題が生まれることもある。指摘がなくなるまで繰り返す

- 修正後、両方のツールを再度実行する

```
再度 markdownlint と textlint を実行してください
```

- 指摘が 0 件になるまで修正と再チェックを繰り返す
- 完了したら校正フェーズは終了。リリースに進む

---

# 6. リリースする（目安: 10 分）

完了の目安: `output/rfp.html` が生成され、ブラウザで表示を確認済み

---

## リリースフェーズでやること

> 校正まで終わった文書を最終成果物として出力する

1. pandoc で Markdown を HTML に変換する
2. ブラウザで変換結果を確認する
3. 問題があれば修正して再変換する

---

## pandoc で HTML に変換する

- Claude Code に指示して変換する

```
src/rfp.md を pandoc で HTML に変換し、output/ に出力してください
```

- pandoc が Markdown を HTML に変換してくれる
- 出力先は `output/rfp.html`

---

## 変換結果を確認する

- ブラウザで `output/rfp.html` を開いて確認する
- 確認のポイント
  - 見出しの階層が正しく反映されているか
  - 表やリストが崩れていないか
  - 全体の読みやすさに問題はないか
- 問題があれば `src/rfp.md` を修正し、再度変換する

---

## 成果物の完成

> ここまでの工程で、構想から始めた RFP が一つの成果物として完成した

- 完成した成果物
  - `input/` — インプット資料
  - `docs/concept.md` — 構想
  - `docs/requirements.md` — 仕様
  - `docs/inspection-log.md` — レビュー記録
  - `src/rfp.md` — RFP 本体
  - `output/rfp.html` — 最終成果物

# ハンズオンの振り返り

---

## 学んだことの整理

- 作業の進め方を工程に分ける
  - 構想 → 仕様 → 執筆 → 校正 → リリース
  - 生成 AI に関係なく、すべての知的作業の基本中の基本
- チャット型生成 AI との違い、Claude Code を「協業者」と見なす
  - 事前情報を共有する
  - 作業工程を共有する
  - 何を依頼し、何を自身で行うかを決める
  - 協業して決めたことを中間文書に落として齟齬を無くす
  - プロジェクトでの共有事項は CLAUDE.md への記録を意識して依頼する
  - プロジェクトごとにどのツールを使うかも事前に共有する

---

## 応用のヒント

- 今回学んだ進め方は RFP に限らず、あらゆる文書作成に応用できる
  - 企画書、報告書、マニュアル、議事録など
- 大事なのはツールではなく「工程を踏む」こと
  - いきなり書き始めず、構想と仕様を先に決める
  - 書いた後に校正ツールで品質を担保する
- Markdown は生成 AI との親和性が高い
  - 生成 AI の入力にも出力にも使われる共通言語
  - 校正ツール、HTML 変換、スライド化など、ツールのエコシステムが豊富
  - プレーンテキストなのでバージョン管理や差分確認がしやすい
- Claude Code は文書作成以外にも使える
  - データの整理、調査、ファイル操作など

# Claude Code セルフハンズオン コンセプト

## タイトル

**「非エンジニア向け Claude Code 入門 — はじめての Claude Code との協業」**

## 対象者

- プログラミング経験がない、または少ない非エンジニア
- 業務でちょっとした文書作成や自動化に生成AIの活用を考えている人
- AI を活用した新しい働き方に興味がある人

## 前提条件

- PC の基本操作ができること（ファイル操作、テキスト入力）
- プログラミングの知識は不要
- Markdown の基本的な書き方を学習済みであること
- シェル環境が利用できること（Git Bash（Git for Windows に付属）。PowerShell やコマンドプロンプトは不可）
- Visual Studio Code（推奨エディタ）がインストール済みであること
- Claude Code がインストール済みであること（事前準備として案内）

## ゴール

1. 文書作成の全工程（構想 → 仕様 → 執筆 → 校正 → リリース）を Claude Code と共に実践できる
2. AI に「丸投げ」するのではなく、人間が主導し AI を道具として使い分ける方法を身につける
3. Unix 哲学に基づく「小さなツールを組み合わせる」作業スタイルを体験する
4. 実際に一つの文書を完成させる

## コンセプト

### 核となる考え方

> **「AI に全部お願い」ではなく「AI と一緒に工程を踏んで仕上げる」**

生成 AI に「○○の文書を書いて」と一発で頼むこともできる。しかしそれでは、意図通りの文書にたどり着けないことが多い。このハンズオンでは、文書作成を明確な工程に分解し、各工程で Claude Code をどう活用するかを学ぶ。

### 文書作成の 5 つの工程

```
構想 → 仕様 → 執筆 → 校正 → リリース
```

| 工程 | やること | Claude Code の役割 |
|---|---|---|
| **構想** | 目的・読者・伝えたいことを整理する | 対話でアイデアを壁打ちし、構成案を生成する |
| **仕様** | 文書の構成・見出し・要件を決める | 構想をもとに仕様書（アウトライン）を作成する |
| **執筆** | 仕様に沿って本文を書く | セクションごとに下書きを生成し、人間が確認・修正する |
| **校正** | 文章の品質をチェックする | textlint, markdownlint 等の校正ツールを実行し、指摘に基づいて修正する |
| **リリース** | 最終確認し、成果物として仕上げる | HTML 変換や最終フォーマット調整を行う |

### 3 つの柱

1. **工程を踏む** — 一発で完成させず、構想から段階的に仕上げる
2. **人間が主導する** — AI は優秀な道具。何を作るか決めるのは自分
3. **ツールを組み合わせる** — Unix 哲学：一つのことをうまくやる小さなツールを Claude Code に使わせる

### Unix スタイルの作業哲学

非エンジニアにとって馴染みのない考え方だが、ハンズオンを通じて自然に体験できる。

- **一つのツールは一つの仕事** — textlint は文章校正、markdownlint は書式チェック、それぞれ専門がある
- **ツールを組み合わせる** — Claude Code がこれらのツールを呼び出し、結果を解釈してくれる
- **人間は判断に集中する** — ツールの使い方は Claude Code に任せ、自分は内容の判断に集中する

### 自己言及的なアプローチ

このハンズオン資料自体が、上記の工程に従って作成されている。資料を読み進めることで「この資料自体がどう作られたか」を追体験できる。

## ハンズオン構成（案）

自分のペースで順番に進められるステップ形式。各ステップは独立した章として資料化する。

| ステップ | 内容 | 対応する工程 | 目安時間 |
|---|---|---|---|
| 1. 環境を整える | ターミナル・Claude Code の起動確認、動作テスト | — | 40 分 |
| 2. 構想する | テーマを決め、Claude Code と壁打ちする | 構想 | 60 分 |
| 3. 仕様を決める | アウトラインを作成し、文書の骨格を固める | 仕様 | 60 分 |
| 4. 執筆する | セクションごとに執筆、Claude Code と推敲する | 執筆 | 30 分 |
| 5. 校正する | markdownlint・textlint で品質チェックと修正 | 校正 | 20 分 |
| 6. 仕上げる | 最終確認、HTML 出力などで成果物を完成させる | リリース | 10 分 |

- 目安時間は参考値。自分のペースで進めてよい
- 各ステップの冒頭に「このステップでやること」「完了の目安」を明記する
- 困らないハンズオンを目指す。分からなければ Claude Code に聞けばよい

## 演習の課題テーマ

**経営支援サービスの Web サイト調達のための RFP（提案依頼書）を作成する。**

架空の経営支援サービス会社が、自社 Web サイトのリニューアルを外部に発注するための RFP を、構想からリリースまでの全工程で仕上げる。

RFP を題材にする理由：

- **構造が明確** — 背景・目的・要件・評価基準など、決まったセクションがあり工程を踏む効果が実感しやすい
- **実務に近い** — 非エンジニアでも業務で触れる機会がある文書形式
- **AI との相性が良い** — 要件の洗い出しや表現の推敲など、Claude Code と対話しながら詰めていく作業に向いている

## 使用ツール

| ツール | 用途 | 備考 |
|---|---|---|
| Claude Code | AI との対話、ファイル操作、ツール実行 | 学習者が指示 → Claude Code が実行 |
| Visual Studio Code | Markdown の編集・プレビュー | 学習者が直接操作 |
| textlint | 日本語文章の校正（冗長表現、表記ゆれ等） | Claude Code が実行 |
| markdownlint | Markdown の書式チェック | Claude Code が実行 |

## 伝えたいメッセージ

- 良い文書は「一発」では生まれない。工程を踏むから質が上がる
- AI は「代筆者」ではなく「共同作業者」。最終判断は常に人間がする
- ツールを使いこなすのに技術知識は要らない。Claude Code が橋渡しをしてくれる
- この作業スタイルは文書作成だけでなく、あらゆる知的作業に応用できる

## 学習のヒント

- エラーが出ても焦らない。エラーは正常な反応であり、Claude Code に「エラーが出た」と伝えれば対処してくれる
- 各ステップを急がず、Claude Code との対話を楽しむ
- うまくいかないときは、指示の出し方を変えてみる。それ自体が学びになる
- ネットワーク接続が必要（Claude Code は API 経由で動作するため）
- 校正ツール（textlint, markdownlint）は事前にインストールしておく
